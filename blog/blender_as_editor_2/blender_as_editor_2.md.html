(insert ../../shared/header.md.html here) **Blender as Editor Devlog #2: Physics**
The rendering path in blender_live_link is actually pretty flexible: if the live-link bridge sends a new mesh or transform, the runtime can usually just scrap the old version and build a fresh one. But physics complicates things. Once an object is in a simulation, its important data might no longer be controlled by Blender. Gravity, collisions, impulses, or even gameplay can move it before the next edit comes through. When Blender sends an update, is it changing the original design, the simulation state, or both? For now, we're keeping the ownership model simple: Blender decides which objects are part of the physics world, and the game engine owns and manages the simulation. If Blender edits an object again, the Blender-authored change takes priority, and the runtime resets/rebuilds the body. **Making the Scene Move**
[Jolt Physics](https://github.com/jrouwe/JoltPhysics) handles the simulation. blender_live_link is intentionally small and avoids build systems like CMake, so Jolt is built as a static library by collecting the required source into a single compilation unit: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ bash default clang -c src/Jolt/jolt_single_file.cpp \ -o ./bin/libjolt.a \ --std=c++20 \ -I src ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ After that, the Jolt setup is mostly adapted from the hello-world sample: initialize the allocator, factory, collision layers, and physics system, then step it every frame. The live-link question starts after that: how does a Blender object become a runtime body, and how does an edit affect those runtime bodies? **Why Physics Changes the Contract**
For rendering, an object update can be treated as replacement data: new mesh, new transform, new storage buffer. With physics, replacements can be more consequential. A body might already exist in the simulation. It might be awake. It might have contacts. It might have velocity. Its transform may no longer match Blender at all. The useful simplification is to make Blender authoritative for edits. If an updated object contains rigid-body data, simply recreate its Jolt body from the latest Blender-authored state. **Rigid Bodies as Data**
The schema changes are minimal. A rigid body is optional object data, which means the presence of the table is the first ownership signal: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ c++ default struct RigidBody { is_dynamic : bool; mass : float; } table Object { // transform, mesh, and other object data omitted rigid_body : RigidBody; } ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ For this version, the runtime assumes a mesh object with rigid-body data should use a convex hull built from the mesh vertices. That keeps the Blender side pleasantly small: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ python default if obj.rigid_body: Object.AddRigidBody(builder, RigidBody.CreateRigidBody( builder, isDynamic = obj.rigid_body.enabled, mass = obj.rigid_body.mass )) ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ The tradeoff is that collision-shape policy currently lives in the runtime. A more complete tool should let Blender express whether an object wants a hull, box, sphere, triangle mesh, or some custom collision representation. **Keeping Simulation State in Sync**
When the game detects that an object has a rigid body, it creates a runtime body description from the Blender data, but defers creation of the Jolt body until the main loop. That allows the game to consume multiple updates before stepping the simulation, and it allows the main loop to manage body lifecycles in a single place: ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ c++ default if (auto object_rigid_body = object->rigid_body()) { game_object.has_rigid_body = true; game_object.rigid_body = (RigidBody){ .is_dynamic = object_rigid_body->is_dynamic(), .mass = object_rigid_body->mass(), .jolt_body = nullptr, }; } ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ When the main loop consumes the updated object, it releases any previous physics body before adding the replacement. ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ c++ default if (state.scene.objects.contains(updated_object_uid)) { Object& existing_object = state.scene.objects[updated_object_uid]; if (existing_object.has_rigid_body) { object_remove_jolt_body(existing_object); } } if (updated_object.has_rigid_body) { object_add_jolt_body(updated_object); } state.scene.objects[updated_object_uid] = updated_object; ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ Right now, the system just destroys and recreates objects. That works fine for early prototyping, but it's not exactly elegant. A more refined approach would distinguish between transform changes, collision-shape updates, explicit simulation resets, and even feedback from the runtime back to Blender. **What This Revealed**
The real takeaway is that live tooling is mostly about ownership. Whether it's meshes, materials, lights, physics, or gameplay components, every new feature asks the same questions again: who owns the source of truth, who owns runtime state, and what messages cross that line? ![](blender_as_editor_2.mov)
